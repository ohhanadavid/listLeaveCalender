# רשימת משימות (Todo List) ל-AI – אפליקציית ניהול לוח חופשות אופליין בריאקט

## הקשר ואילוצים שסוכמו
- אפליקציית React אחת, רצה אופליין לחלוטין (ללא שרת/backend), על מכשיר/דפדפן משותף אחד (קיוסק).
- כל הנתונים נשמרים מקומית בדפדפן (IndexedDB) ושורדים בין טעינות דף.
- גורם מאשר יחיד, מוגן בסיסמה/PIN בתוך האפליקציה (שימו לב: זו לא אבטחה אמיתית, רק חסם שימוש לקיוסק משותף).
- שאר המשתמשים (עד 10 עובדים, בלי מחלקות) יכולים רק להגיש בקשות, לא לשנות את הלוח עצמו.
- בנייה אוטומטית – שני מנגנונים אפשריים לכל עובד בנפרד, ניתנים לבחירה: (א) דפוס מחזורי קבוע (X ימי עבודה + Y ימי חופשה, חוזר אוטומטית), (ב) טווח תאריכים שמוגדר ידנית.
- חופשות לפי שעות – שני דברים נפרדים: (1) בקשת חופשה חלקית ביום עם שעת התחלה/סיום, (2) רישום שעות כניסה/יציאה רגילות (נוכחות).
- בקרת מינימום אנשים בעבודה – רק התרעה/אזהרה ויזואלית בלוח, לא חוסמת פעולות בפועל.
- לוח שנה ויזואלי שמציג הכל + תאריכים עבריים חשובים (חגים, מועדים).
- אף יום אינו "יום חופש" מעצם טבעו – כולל שבתות וחגים. הסטטוס של כל יום נקבע אך ורק לפי הדפוס/הבקשות שהוגדרו, בלי לוגיקה מיוחדת לשבת/חג.
- ללוח יש תאריך התחלה ותאריך סיום גלובליים, משותפים לכולם (לדוגמה 22.06–13.08), עם אפשרות להאריך את הטווח מאוחר יותר.
- האחסון הוא קובץ פשוט ונפרד (JSON) על הדיסק – לא מסד נתונים בדפדפן (לא IndexedDB) ולא שרת חיצוני. השמירה אוטומטית (לא ידנית) – ראו הערה טכנית בשלב 8.
- המאשר מגדיר מראש את רשימת העובדים (אין הרשמה עצמית). לכל עובד נקבע צבע – אקראי או נבחר ידנית – והצבע הזה מופיע במשבצת התאריך בלוח בכל יום שהעובד נמצא בו בחופשה.
- כשבקשת עובד מתנגשת עם דפוס אוטומטי קיים – אין הכרעה אוטומטית, המערכת רק מסמנת את ההתנגשות והמאשר מחליט ידנית.
- 100% אופליין, בלי שום יוצא מן הכלל: אסור לטעון ספריות/פונטים/אייקונים מ-CDN, אסורה כל קריאת API חיצונית (אנליטיקס, פונטים מגוגל וכו'). כל הספריות (FullCalendar, @hebcal/core וכו') מותקנות כחבילות npm ונכללות בתוך ה-bundle המקומי בזמן build.

## ערימת טכנולוגיות מומלצת
- [ ] React + Vite
- [ ] אחסון: קובץ JSON יחיד על הדיסק, נשמר אוטומטית בכל שינוי דרך File System Access API (`showOpenFilePicker` לבחירת הקובץ פעם אחת בכניסה לאפליקציה, ואז `createWritable`/`write` בכל עדכון). דורש דפדפן מבוסס Chromium (Chrome/Edge) – לא עובד בפיירפוקס/ספארי
- [ ] הערה טכנית חשובה: ה-API לא מאפשר "לזכור" את הקובץ בין הפעלות דפדפן בלי לאחסן handle, ואחסון handle דורש IndexedDB. כדי לעמוד בדרישה "בלי DB", הפתרון הוא לבקש מהמשתמש לבחור את הקובץ פעם אחת בכל פתיחה מחדש של הדפדפן/האפליקציה (לא בכל שמירה) – אחרי הבחירה, כל השינויים נשמרים אוטומטית בלי לחיצות נוספות
- [ ] גיבוי נוסף לכל מקרה: כפתור "ייצוא לקובץ חדש" ידני (Blob + `<a download>`), למקרה שהדפדפן לא תומך ב-API או לגיבוי נפרד
- [ ] לוח שנה ויזואלי: FullCalendar (react wrapper) או react-big-calendar
- [ ] תאריכים עבריים וחגים: @hebcal/core (עובד לגמרי אופליין, בלי תלות ברשת)
- [ ] ניהול state: Context API או Zustand (מספיק לגודל הזה, אין צורך ב-Redux)
- [ ] שקלו vite-plugin-pwa כדי שהאפליקציה תיטען גם ללא אינטרנט לחלוטין אחרי טעינה ראשונה

## עיצוב (Design System)
הקשר: מסך שולחני קבוע (קיוסק), כיוון עיצובי מינימליסטי/עסקי-נקי, ממשק בעברית (RTL).

### פלטת צבעים
UI (ניטרלי + accent מצומצם, לא לבלבל עם צבעי העובדים בלוח):
- [ ] רקע: `#FAF9F6` (כמעט-לבן חמים, לא לבן קר)
- [ ] משטחים (כרטיסים/טפסים): `#FFFFFF`
- [ ] טקסט ראשי: `#1F2933`
- [ ] טקסט משני: `#5C6670`
- [ ] קווי מתאר/הפרדה: `#E2E5E8`
- [ ] accent ראשי (פעולות עיקריות + אינדיקציית "מצב מאשר פעיל"): `#0F6B5C` (ירוק-טיל כהה, מקצועי, לא כחול קלישאתי)
- [ ] accent אזהרה (ימים מתחת למינימום אנשים): `#C2780C` (כתום-ענבר), בשימוש מצומצם
- [ ] accent קונפליקט/שגיאה: `#B3261E`, שמור אך ורק להתנגשויות ושגיאות כתיבה לקובץ

פלטת עובדים (נפרדת, קטגורית, לזיהוי בלוח):
- [ ] עד 10 צבעים נבדלים מתוך פלטה "ידידותית לעיוורי צבעים" (לדוגמה משפחת Okabe-Ito), לא צבעי UI שמשמשים גם לאזהרות/שגיאות
- [ ] חשוב: לא להסתמך רק על צבע להבחנה בין עובדים – כל סימון בלוח כולל גם תווית טקסט קצרה (ראשי תיבות/שם מקוצר), כדי שזה יהיה ברור גם למי שמתקשה להבחין בצבעים

### טיפוגרפיה
- [ ] גופן עברי קריא: Heebo או Assistant – מותקן כחבילת npm/קובץ סטטי מקומי, **לא** טעינה מ-Google Fonts CDN (דרישת אופליין מוחלט)
- [ ] היררכיית גדלים: כותרת מסך 24–28px, כותרות משנה 18–20px, גוף טקסט 14–16px, מספרים/נתוני לוח 13–14px
- [ ] משקלים: Regular לטקסט שוטף, Medium/SemiBold לכותרות, תוויות ופעולות

### Layout
- [ ] `dir="rtl"` על כל האפליקציה, כולל בדיקת יישור נכון של אייקונים/חיצים/תאריכים
- [ ] עיצוב מותאם למסך גדול קבוע (1920×1080 ו-1366×768), לא בעדיפות ראשונה למובייל – אך לא "להישבר" במסך קטן יותר
- [ ] הלוח (calendar) הוא מרכז הכובד של המסך הראשי, תופס את רוב השטח; פס עליון דק וקבוע לפעולות: כניסה/יציאה ממצב מאשר, badge בקשות ממתינות, כפתור "החלף קובץ שמירה"
- [ ] כש-isApproverMode פעיל: אינדיקציה ויזואלית קבועה וברורה (פס עליון בצבע ה-accent הראשי + תווית "מצב מאשר פעיל"), כדי שלעולם לא יהיה ספק באיזה מצב נמצאים

### אלמנט מזהה (signature)
- [ ] בכל תא יום בלוח שיש בו עובד בחופשה: "רצועת" צבע קטנה בתחתית התא (לא צביעת כל התא) בצבע העובד + ראשי תיבות, כך שריבוי עובדים באותו יום נשאר קריא ולא הומוגני

### נגישות ואיכות בסיסית
- [ ] ניגודיות טקסט/רקע עומדת ב-WCAG AA לפחות
- [ ] focus ויזואלי ברור על כל שדה/כפתור בר-מיקוד (חשוב במיוחד לשדה ה-PIN)
- [ ] גודל אזורי לחיצה נוח לשימוש במסך מגע/עכבר בקיוסק (לא כפתורים זעירים)

## שלב 0 – הקמת פרויקט
- [ ] init פרויקט Vite + React + TypeScript
- [ ] מבנה תיקיות: components/, data/ (שכבת קריאה/כתיבה לקובץ JSON, ראו שלב 8), hooks/, types/
- [ ] הגדרת טיפוסי TypeScript למבנה ה-JSON: employees, leavePatterns, leaveRequests, attendanceLogs, settings

## שלב 1 – מבנה קובץ ה-JSON
כל הנתונים נשמרים באובייקט JSON אחד עם המפתחות הבאים:
- [ ] employees: מערך { id, name, active, color }
- [ ] leavePatterns (לדפוס מחזורי): מערך { id, employeeId, workDays, leaveDays, startDate, isActive }
- [ ] leaveRequests: מערך { id, employeeId, date, type ('full'|'hours'), startTime?, endTime?, status ('pending'|'approved'|'rejected'), source ('auto'|'manual'|'request') }
- [ ] attendanceLogs (נוכחות שעתית, נפרד מחופשה): מערך { id, employeeId, date, checkIn, checkOut }
- [ ] settings: { approverPin, minStaffPerDay, boardStartDate, boardEndDate }

## שלב 2 – הגנת מאשר (PIN)
- [ ] מסך כניסה למצב "מאשר" עם הקלדת PIN
- [ ] state גלובלי isApproverMode + כפתור "יציאה ממצב מאשר" (וכדאי גם timeout אוטומטי לחוסר פעילות, כי זה קיוסק משותף)
- [ ] רק ב-isApproverMode=true מותר: אישור/דחיית בקשות, עריכה/מחיקה של ימי חופשה, עריכת דפוסים אוטומטיים, שינוי minStaffPerDay
- [ ] בלי isApproverMode: אפשר רק להגיש בקשת חופשה חדשה

## שלב 2.5 – ניהול עובדים (רק במצב מאשר)
- [ ] מסך לניהול רשימת העובדים: הוספה, עריכה, השבתה (לא מחיקה – כדי לא לאבד היסטוריה)
- [ ] בחירת צבע לכל עובד בעת היצירה: כפתור "הגרל צבע" (אקראי מתוך פלטה קבועה) או בחירה ידנית מ-color picker
- [ ] אזהרה (לא חסימה) אם שני עובדים פעילים קיבלו צבע זהה או דומה מדי

## שלב 3 – בנייה אוטומטית של הלוח
- [ ] טופס "דפוס מחזורי": בחירת עובד + ימי עבודה רצופים + ימי חופשה רצופים + תאריך התחלה → אלגוריתם שמייצר רשומות leaveRequests (status='approved', source='auto') מתאריך ההתחלה ועד boardEndDate (מתוך settings), ספירה רציפה של ימים בלי דילוג על שבתות/חגים
- [ ] טופס "טווח ידני": בחירת עובד + טווח תאריכים (בתוך גבולות הלוח) → יצירת רשומות חופשה ישירות
- [ ] כשמאריכים את boardEndDate בהגדרות: להריץ מחדש את הבנייה האוטומטית לכל הדפוסים הפעילים כדי למלא את הטווח החדש
- [ ] לוגיקת עדכון דפוס קיים: לאפשר לבחור בין "לעדכן רק מכאן והלאה" לבין "למחוק ולבנות מחדש את כל העתיד"
- [ ] ולידציה נגד התנגשות כפולה לאותו עובד באותו תאריך

## שלב 4 – בקשות חופשה (עובדים)
- [ ] טופס הגשת בקשה: עובד, תאריך/טווח, סוג (יום מלא / שעות עם שעת התחלה־סיום)
- [ ] בקשה חדשה נכנסת כ-status='pending'
- [ ] אם הבקשה מתנגשת עם רשומה קיימת מסוג source='auto' לאותו עובד/תאריך – לסמן זאת בבירור במסך הבקשות הממתינות (לא לחסום ולא להכריע אוטומטית)
- [ ] מסך למאשר: רשימת בקשות ממתינות + אישור/דחייה. במקרה התנגשות – האישור דורס את הרשומה האוטומטית הקיימת לאותו תאריך
- [ ] badge עם מספר הבקשות הממתינות, מוצג באופן קבוע (לדוגמה ליד כפתור הכניסה למצב מאשר או בתפריט הראשי), כדי שהמאשר יראה מיד שיש בקשות לטיפול בלי להיכנס למסך ייעודי

## שלב 5 – רישום נוכחות שעתית
- [ ] טופס להזנת שעת כניסה ושעת יציאה ביום נתון לעובד (נפרד לגמרי מבקשות חופשה)
- [ ] תצוגת היסטוריית נוכחות לפי עובד/תאריך

## שלב 6 – בקרת מינימום אנשים
- [ ] חישוב לכל יום בלוח: כמה עובדים פעילים (לא בחופשה) באותו יום
- [ ] אם המספר < minStaffPerDay → סימון ויזואלי (למשל צבע/אייקון אזהרה) על אותו יום, בלי לחסום פעולות
- [ ] מסך הגדרות עם שדות: minStaffPerDay, boardStartDate, boardEndDate (כולל אפשרות לערוך/להאריך את boardEndDate בכל שלב)

## שלב 7 – לוח שנה ויזואלי
- [ ] תצוגת חודש/שבוע ב-FullCalendar; כל יום שעובד בחופשה בו מציג רצועת צבע + ראשי תיבות (לפי ה-design system), לא צביעת כל התא
- [ ] שכבת "ימי אזהרת מינימום" (משלב 6)
- [ ] הצגת תאריך עברי מתחת לכל יום (@hebcal/core)
- [ ] סימון חגים/מועדים עבריים בלוח
- [ ] סינון/הדגשה לפי עובד בודד

## שלב 8 – שמירה וטעינה מקובץ (מנגנון האחסון הראשי, אוטומטי)
- [ ] בכניסה לאפליקציה: אם אין handle פעיל לקובץ – לבקש מהמשתמש לבחור קובץ קיים (`showOpenFilePicker`) או ליצור חדש (`showSaveFilePicker`). זו מגבלת אבטחה של הדפדפן (לא ניתן לעקיפה) – לכן זה קורה בכל פתיחה מחדש של הדפדפן/האפליקציה, לא רק בפעם הראשונה
- [ ] כל הנתונים מוחזקים ב-state בזיכרון בזמן השימוש באפליקציה
- [ ] בכל שינוי ב-state (הוספת עובד, בקשה, אישור וכו') – כתיבה אוטומטית לקובץ דרך ה-handle (`createWritable` → `write` → `close`), בלי כל פעולת משתמש נוספת
- [ ] במסך ההגדרות: כפתור "החלף קובץ שמירה" שפותח מחדש את תיבת הבחירה ומאפשר לעבור לקובץ אחר (אין שדה נתיב טקסט – הדפדפן לא חושף נתיבים ל-JS, רק תיבת בחירה)
- [ ] טיפול בשגיאת כתיבה (לדוגמה הקובץ נמחק/הוסרה הרשאה) – הודעה ברורה למשתמש, לא כשל שקט
- [ ] נפילה חזרה (fallback) לדפדפנים שלא תומכים ב-File System Access API: כפתור "ייצוא לקובץ" ידני בלבד, עם אזהרה שהשמירה במצב הזה אינה אוטומטית

## שלב 9 – בדיקת קבלה לאופליין מוחלט
- [ ] לפני מסירה: לנתק לגמרי את החיבור לאינטרנט (Wi-Fi/כבל כבויים) ולוודא שהאפליקציה עולה ועובדת במלואה – כולל לוח שנה, תאריכים עבריים, שמירה/טעינה מקובץ
- [ ] לבדוק ב-DevTools (לשונית Network) שאין אף קריאת רשת יוצאת בזמן שימוש רגיל באפליקציה
